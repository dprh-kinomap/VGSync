1. Git-Grundlagen

Git ist ein verteiltes Versionskontrollsystem. Damit kannst du:

  • Änderungen an deinen Dateien (Code, Konfig, etc.) versionieren und verwalten.
  • Zu älteren Ständen zurückkehren, wenn etwas schiefgeht.
  • Parallele Entwicklungszweige (Branches) anlegen, ohne den stabilen Code zu gefährden.
  • Änderungen mit anderen teilen und synchronisieren (z. B. über GitHub).

2. Installation & Erste Konfiguration

  1. Git installieren
    • Windows: Git für Windows
    • macOS: brew install git
    • Linux (Ubuntu): sudo apt-get install git
  2. Name & E-Mail konfigurieren 
    git config --global user.name "Dein Name"
git config --global user.email "deine.email@example.com"

    Diese Angaben erscheinen in deinen Commits.

3. Grundlegender Workflow

Der typische Ablauf beim Arbeiten mit Git:

  1. Neues Repository initialisieren (oder klonen) 
    git init
# oder falls du ein bestehendes Repo klonen willst:
git clone
  2. Änderungen machen – Dateien bearbeiten, erstellen, löschen
  3. Status prüfen 
    git status
  4. Änderungen zum Commit hinzufügen (stagen) 
    git add datei.txt
git add .
  5. Commit erstellen 
    git commit -m "Sinnvolle Beschreibung"
  6. Versionen anzeigen 
    git log
git log --oneline --decorate --graph --all
  7. Branching / Merging – Parallele Entwicklungszweige nutzen
  8. Push & Pull – mit Remote (z. B. GitHub) synchronisieren

4. Wichtige Git-Befehle (Tabelle)

Befehl Beschreibung
git init Erzeugt ein neues lokales Git-Repository.
git clone <URL> Lädt ein bestehendes Repository (z. B. von GitHub) auf den lokalen Rechner.
git status Zeigt den aktuellen Zustand: welche Dateien sind geändert, ob Commits ausstehen usw.
git add <Datei> Markiert eine Datei für den nächsten Commit. git add . nimmt alle geänderten Dateien auf.
git commit -m "Nachricht" Erstellt einen neuen Commit mit den vorgemerkten (gestagten) Dateien.
git log Zeigt eine Liste aller Commits. Mit --oneline --graph --decorate --all wird es schöner.
git branch Listet alle lokalen Branches auf. Ein Stern * markiert den aktuellen Branch.
git checkout -b <branch> Erstellt einen neuen Branch und wechselt direkt hinein.
git checkout <branch> Wechselt zu einem bestehenden Branch oder Commit (z. B. Tag).
git merge <branch> Merged einen anderen Branch in den aktuellen Branch.
git branch -d <branch> Löscht einen Branch (wenn gemergt). -D erzwingt das Löschen.
git tag <version> Markiert den aktuellen Commit mit einer Versionsnummer (z. B. v1.0).
git push origin <branch|tag> Lädt einen bestimmten Branch oder Tag ins Remote-Repository (z. B. GitHub).
git pull Holt Änderungen vom Remote-Repository und merged sie in den lokalen Stand.
git stash „Parkt“ aktuelle Änderungen, ohne zu committen. Mit git stash pop holst du sie wieder.
git revert HEAD Macht den letzten Commit rückgängig, erzeugt aber einen neuen Commit (sicherer in Teamprojekten).
git reset --hard <Commit-Hash> Setzt den aktuellen Branch auf einen älteren Commit zurück und verwirft neuere Commits (vorsichtig!).
git restore --staged <Datei> Nimmt die Datei wieder aus dem Staging-Bereich (vor dem Commit) heraus.
git checkout -- <Datei> Setzt eine Datei auf die letzte gespeicherte Version zurück (vor ungespeicherten Änderungen).

5. Beispiele & Szenarien

Beispiel A: Einen neuen Branch erstellen & testen
  1. Ausgangssituation: Du befindest dich in main und hast eine stabile Version (z. B. v3.243).
  2. Neuen Branch erstellen:
    git checkout -b open-elavation
    Jetzt bist du in open-elavation.
  3. Änderungen machen & committen:
    git add .
git commit -m "Neues Feature open-elavation umgesetzt"
  4. Version prüfen:
    git log --oneline --decorate --graph --all
    Du siehst, dass open-elavation jetzt neue Commits hat.
  5. (Optional) Neue Version taggen:
    git tag v3.25
    Damit hast du einen Schnappschuss deiner Arbeit in v3.25.
  6. Branch zurück in main mergen (falls alles gut ist): 
    git checkout main
git merge open-elavation
  7. Branch löschen (falls nicht mehr nötig):
    git branch -d open-elavation
Beispiel B: Zurück zur alten Version
  1. Du hast bereits ein Tag namens v3.243, das deine stabile Version markiert.
  2. Wechsel zu v3.243:
    git checkout v3.243
    Jetzt bist du in einer älteren Version (detached HEAD-Modus).
  3. Nicht änderbar?
    In diesem Modus kannst du zwar schauen, aber nicht dauerhaft committen. Falls du trotzdem Änderungen machen willst: 
    git checkout -b hotfix-based-on-3.243
    Jetzt hast du einen neuen Branch basierend auf v3.243.
Beispiel C: „Falsches“ Tag löschen oder neu erstellen
  1. Tag löschen (lokal):
    git tag -d v3.25
  2. Tag erneut setzen:
    git checkout open-elavation
git tag v3.25
    Jetzt zeigt v3.25 auf den aktuellen Commit von open-elavation.
  3. Tag hochladen (GitHub):
    git push origin v3.25

6. Tipps & Best Practices

  • Committe früh und oft: So hast du viele kleine Schritte, zu denen du zurückkehren kannst.
  • Aussagekräftige Commit-Botschaften: Was genau wurde geändert? Warum?
  • Nutze Branches für neue Features oder Experimente: Halte main stabil.
  • Tags für Versionen: Markiere wichtige Meilensteine (z. B. v1.0, v2.0).
  • Reset vs. Revert: In Teamprojekten lieber revert statt reset verwenden, um keine fremden Commits zu vernichten.
  • GitHub / Remote: Push deine Commits regelmäßig, damit nichts verloren geht.
Hinweis: Git hat viele weitere fortgeschrittene Befehle (z. B. cherry-pick, rebase, bisect), die hier aus Platzgründen nur am Rande erwähnt sind.

Einfache Schritt-für-Schritt-Anleitung (Hauptversion & Testversion)

Ziel: Wir haben eine stabile Hauptversion (z. B. main) und wollen eine Testversion erstellen. Wenn wir zufrieden sind, machen wir daraus eine neue Hauptversion. Wenn wir unzufrieden sind, verwerfen wir die Testversion.

  1. Hauptversion herausfinden 
    git branch

    Der aktive Branch ist mit * markiert. Meist heißt er main oder master.

  2. Neue Testversion (Branch) erstellen & direkt wechseln 
    git checkout -b testversion
    • Erstellt einen neuen Branch namens testversion.
    • Wechselt automatisch dorthin.
  3. Änderungen machen & committen 
    git add .
git commit -m "Funktion in testversion umgesetzt"

    Wiederhole diesen Schritt beliebig oft, während du deine Testversion weiterentwickelst.

  4. Testversion übernehmen (wenn zufrieden)
    Du möchtest jetzt die Testversion in deine Hauptversion übernehmen. 
    git checkout main
git merge testversion
git branch -d testversion
    • checkout main: Zurück zur Hauptversion wechseln.
    • merge testversion: Die Änderungen von testversion in main übernehmen.
    • branch -d testversion: Den Test-Branch löschen, da er jetzt in main steckt.
  5. Testversion verwerfen (wenn unzufrieden)
    Du möchtest die Testversion komplett löschen, ohne sie zu übernehmen. 
    git checkout main
git branch -D testversion

    Damit wird der Branch testversion (und alle Commits, die nur dort existierten) verworfen. Vorsicht: Dadurch gehen alle Änderungen verloren, die nicht in main gemergt wurden!

  6. Zwischen Hauptversionen wechseln
    Wenn du mehrere Hauptversionen als Branches oder Tags hast, wechsle einfach mit: 
    git checkout

    Beispiel: git checkout main wechselt zu main, git checkout v3.243 wechselt zum Tag v3.243.
