1. Was ist Git und warum brauche ich das?

Git ist ein Werkzeug zur Versionsverwaltung deines Codes. Stell es dir wie ein Rückgängig- und Wiederholen-Knopf für dein gesamtes Projekt vor, nur viel mächtiger.

  • Du kannst jederzeit den Zustand deines Projekts (Quellcode) in "Schnappschüssen" festhalten.
  • Falls etwas schiefläuft, kannst du zu einer älteren, funktionierenden Version zurückspringen.
  • Mit GitHub oder anderen Plattformen kannst du deinen Code außerdem leicht in der Cloud sichern und mit anderen teilen.

2. Vorbereitung: Git installieren

Wenn du Git noch nicht installiert hast, hole das jetzt nach:

  • Windows: Lade den Installer unter https://git-scm.com/download/win herunter und führe ihn aus.
  • macOS: Mit Homebrew: brew install git oder lade es von git-scm.com/download/mac herunter.
  • Linux (z.B. Ubuntu): sudo apt-get update, sudo apt-get install git

Teste anschließend die Installation:

git --version

Wenn eine Versionsnummer erscheint, ist alles korrekt installiert.

3. Erste Schritte im Projekt

Wir gehen davon aus, dass du bereits ein bestehendes Python-Projekt (oder auch ein leeres Verzeichnis für ein neues Projekt) hast.

  1. In das Verzeichnis des Projekts wechseln.
    Beispiel: cd /pfad/zu/deinem/python-projekt
  2. Git initialisieren.
    git init
    Das erzeugt im Hintergrund einen versteckten Ordner namens .git, der all deine Versionsinformationen speichert.
Hinweis: Sobald du git init gemacht hast, wird dein Verzeichnis zum "Git-Repository". Das heißt, Git überwacht ab jetzt deine Dateien (sobald du sie hinzufügst) und kann Versionen davon speichern.

4. .gitignore verstehen und erstellen

In einer Datei namens .gitignore stehen alle Dateien oder Ordner, die Git nicht in die Versionsverwaltung aufnehmen soll. Das ist wichtig, damit zum Beispiel dein virtuelles Python-Umfeld (venv), automatisch erzeugte Cache-Dateien oder sensible Daten nicht mit versioniert werden.

Typische Einträge für ein Python-Projekt:

__pycache__/
*.py[cod]
venv/
.venv/
build/
dist/
*.egg-info/
.env

Erstelle eine neue Textdatei mit dem Namen .gitignore in deinem Projektordner und kopiere obenstehende Zeilen hinein (passe sie bei Bedarf an).

5. Versionen erfassen: git add und git commit

  1. Änderungen "stagen" (mit git add)
    Wenn du an deinen Dateien arbeitest, ändert Git noch nichts automatisch. Erst wenn du sagst git add <datei>, landen die Änderungen im sogenannten "Staging-Bereich". Das bedeutet, du markierst sie für den nächsten Schnappschuss (Commit).

    Möchtest du alle geänderten Dateien auf einmal hinzufügen, kannst du schreiben: git add .
  2. Schnappschuss erstellen (mit git commit)
    git commit -m "Kurze Beschreibung"
    Damit erstellst du jetzt wirklich den Versions-Schnappschuss. Die Nachricht (in Anführungszeichen) sollte kurz und prägnant beschreiben, was sich geändert hat, z. B. "Feature X hinzugefügt" oder "Bug Y behoben".
Warum zwei Schritte (add und commit)?
Das ermöglicht dir, genau zu kontrollieren, welche Änderungen in den nächsten Versions-Schnappschuss gehören und welche nicht.

6. Änderungen ansehen und überprüfen

  • Status überprüfen:
    git status
    So siehst du, welche Dateien gerade verändert oder noch nicht "gestaged" sind.
  • Unterschiede anzeigen:
    git diff
    Zeigt dir Zeile für Zeile an, was sich seit dem letzten Commit verändert hat.
  • Bisherige Commits (Verlauf) anschauen:
    git log
    Listet alle bisherigen Commits mit Autor, Datum und Nachricht auf.

7. Vergangene Versionen anzeigen und wiederherstellen

Manchmal möchtest du eine alte Version deines Projekts anschauen oder sogar dorthin zurückkehren.

  • Alte Version "nur ansehen":
    git checkout <Commit-Hash>
    Damit bewegst du dich temporär in einen älteren Zustand. Das wird "detached HEAD" genannt. Wenn du wieder zu deinem aktuellen Hauptzweig willst, tippst du: git checkout main.
  • Version dauerhaft zurücksetzen:
    • Revert (empfohlen, da sicherer): git revert <Commit-Hash>
      Das erstellt einen neuen Commit, der die Änderungen von damals rückgängig macht, ohne den Verlauf zu zerstören.
    • Reset (Vorsicht!): git reset --hard <Commit-Hash>
      Hier wird der Projektstand auf den angegebenen Commit zurückgespult. Alle commits, die danach kamen, sind lokal weg (entweder überschrieben oder verloren), was in Teamprojekten für Verwirrung sorgen kann.

8. Arbeiten mit Branches (Kurzüberblick)

Ein Branch ist wie ein Entwicklungszweig. Standardmäßig hast du den main-Branch. Für neue Features oder Experimente legst du am besten eigene Branches an, damit du deinen Hauptzweig stabil hältst. 

git checkout -b feature/neues_feature
# wechselt automatisch in den neuen Branch 'feature/neues_feature'

Zurück zum main-Branch: 

git checkout main

Um deinen Feature-Branch am Ende in main zu integrieren, führst du einen Merge durch: 

git checkout main
git merge feature/neues_feature

9. Projekt auf GitHub hochladen

Möchtest du dein lokales Git-Projekt auf GitHub speichern, gehe so vor:

  1. Neues Repository auf GitHub erstellen:
    Gehe zu github.com, melde dich an und klicke auf New. Vergib einen Namen und lege fest, ob es öffentlich oder privat sein soll.
  2. Lokales Projekt mit dem GitHub-Repo verknüpfen:
    Im Terminal (immer noch in deinem Projektordner): 
    git remote add origin https://github.com/deinBenutzername/deinRepositoryName.git
git push -u origin main
    Damit verbindest du deinen lokalen main-Branch mit dem sogenannten Remote namens origin (GitHub).
    Das -u stellt eine "Tracking-Verbindung" her, sodass du in Zukunft nur git push tippen musst.
  3. Änderungen hochladen:
    Nach neuen Commits genügt git push, um deinen Code auf GitHub zu übertragen.

10. Alltäglicher Ablauf im Schnelldurchlauf

  1. Aktuellsten Stand holen (wenn du mit anderen zusammenarbeitest):
    git pull
  2. Code bearbeiten, Dateien verändern.
  3. Status prüfen:
    git status
  4. Änderungen stagen:
    git add . (oder gezielt bestimmte Dateien)
  5. Commit erstellen:
    git commit -m "Beschreibender Text"
  6. Änderungen (falls nötig) hochladen:
    git push
Wichtig: Regelmäßig und in kleinen Schritten committen. So bleibt deine Historie übersichtlich und du kannst leichter Fehler finden oder rückgängig machen.

Damit hast du jetzt eine verständliche Grundlage, um Git in deinem Python-Projekt zu nutzen – und zugleich eine Möglichkeit, den Code sicher auf GitHub zu verwalten.
